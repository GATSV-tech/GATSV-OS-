# GATSV OS Core v1 — Architecture

## Core Event Loop

Every piece of work follows one path without exception:

```
Inbound → Normalize → Classify → Route → Act → Approve → Execute → Log → Surface
```

Nothing bypasses this loop. Nothing acts without logging. No agent skips the schema.

## Database Schema (Slice 2)

Five tables: `events`, `entities`, `actions`, `approvals`, `memories`, `health_logs`.
Full schema in `db/migrations/001_initial.sql`.

## Agent Roles

| Agent | Slice | Responsibility |
|-------|-------|----------------|
| Gatekeeper | 5 | Normalize inbound → event schema, dedup, create entity |
| Router | 6 | Classify event → bucket + priority via Claude |
| Operator | 7 | Execute safe actions, queue risky ones for approval |
| Reporter | 9 | Daily Slack digest, on-demand summaries |
| Auditor | 10 | Cost tracking, anomaly detection, health logs |

## Interface Surfaces

- **v1:** Slack (approvals, alerts, summaries, commands) — interface layer only, not core logic
- **v2/vNext:** Lightweight web dashboard
- All surfaces are replaceable; agent logic is never coupled to a specific channel

## Inbound Sources (v1)

- **Email:** Postmark inbound webhook → `POST /inbound/email`
- **Forms:** Tally webhook → `POST /inbound/form`

## Observability Contract

Every agent action must:
1. Write a row to `actions` with `token_input`, `token_output`, `usd_cost`, `duration_ms`
2. Write a row to `health_logs` with service name and event type
3. Never fail silently — errors are logged to `health_logs` with full context

## Routing Buckets

| Bucket | Meaning |
|--------|---------|
| `sales` | New inquiry, pricing question, wants to work together |
| `delivery` | Existing client, project update, deliverable question |
| `support` | Something broken, complaint, urgent |
| `founder_review` | Ambiguous, high-stakes, or Router confidence < 0.75 |
| `noise` | Spam, irrelevant, automated system mail |
